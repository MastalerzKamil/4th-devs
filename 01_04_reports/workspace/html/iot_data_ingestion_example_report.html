Quality signals & next actions

Data quality signals

In practice, the most useful ingestion report separates transport reliability (did the event arrive?) from data validity (is the event usable?).

Signal Interpretation Status Observed
Error rate API/ingest failures; often auth, throttling, or transient infra Elevated 10%
Duplicates Retry storms, at-least-once delivery without idempotency Elevated 10%
Schema failures Firmware drift; missing fields; incompatible encoders Elevated 10%
P95 latency End-to-end ingestion delay (device publish → durable storage) OK 2.1s

Recommended next actions

  1. Make ingest idempotent: add a stable event key (device + metric + timestamp + sequence) and dedupe downstream.
  2. Quarantine invalid payloads: route schema failures to a DLQ with replay after fixes.
  3. Segment errors: break down 4xx vs 5xx and slice by gateway_id, region, fw_version.
  4. Define stage SLOs: edge (MQTT) → API → bus → processing → storage so latency regressions localize quickly.
